Estamos diseñando un sistema organizacional multi-agente enfocado en observabilidad, comunicación estructurada y coordinación emergente entre agentes inteligentes.

━━━━━━━━━━━━━━━━━━
FILOSOFÍA GENERAL
━━━━━━━━━━━━━━━━━━

El sistema NO debe pensarse como:
- una aplicación de chat,
- un framework clásico de agentes,
- un sistema de prompts gigantes,
- ni un workflow engine tradicional.

El sistema debe pensarse como:
- un runtime organizacional,
- un ecosistema de agentes autónomos,
- un protocolo de coordinación distribuida,
- y una plataforma de observabilidad organizacional en tiempo real.

El objetivo principal NO es controlar agentes.
El objetivo principal es:
- visualizar,
- observar,
- rastrear,
- y entender
cómo agentes autónomos colaboran organizacionalmente.

━━━━━━━━━━━━━━━━━━
QUÉ ES UN AGENTE
━━━━━━━━━━━━━━━━━━

Un agente NO es un LLM.

Un agente representa:
- una identidad organizacional,
- un rol persistente,
- una responsabilidad,
- o una función dentro de una organización artificial.

Ejemplos:
- Frontend Lead
- Backend Architect
- UI Designer
- QA Reviewer
- Research Specialist

El agente es persistente organizacionalmente.

El LLM NO es el agente.

El LLM es solamente:
- el runtime cognitivo temporal
que opera al agente.

━━━━━━━━━━━━━━━━━━
CONTINUIDAD ORGANIZACIONAL
━━━━━━━━━━━━━━━━━━

La continuidad del sistema NO es cognitiva.
Es organizacional.

Ejemplo:

Frontend Agent:
- crea instrucciones,
- responde tareas,
- mantiene relaciones organizacionales.

Si el LLM que operaba ese agente desaparece:
otro LLM puede continuar operando la misma identidad organizacional.

Ese nuevo runtime:
- NO tendrá exactamente el mismo contexto interno,
- NO conservará razonamiento completo,
- y probablemente tendrá dificultades iniciales.

Pero:
la identidad organizacional continúa existiendo.

Esto es intencional.

El sistema preserva:
- continuidad organizacional.

NO:
- conciencia,
- ni estado cognitivo perfecto.

━━━━━━━━━━━━━━━━━━
AGENTES AUTÓNOMOS
━━━━━━━━━━━━━━━━━━

Los agentes deben ser altamente autónomos.

El backend NO controla:
- decisiones,
- workflows internos,
- razonamiento,
- ni planificación.

El backend solamente facilita:
- comunicaciones,
- eventos,
- sincronización,
- y observabilidad.

Los agentes:
- deciden,
- delegan,
- crean relaciones,
- y organizan trabajo por sí mismos.

━━━━━━━━━━━━━━━━━━
LIMITES DE AUTONOMÍA
━━━━━━━━━━━━━━━━━━

Los límites de autonomía NO deben ser rígidos inicialmente.

Sin embargo:
los agentes NO deben:
- ignorar políticas organizacionales,
- ni realizar tareas fuera de su rol.

Ejemplo:
Frontend Agent NO debe hacer trabajo de Backend.

Si recibe una instrucción mixta:
debe:
- completar solamente la parte correspondiente a frontend,
- y aclarar qué falta completar.

Los agentes SI pueden:
- crear relaciones nuevas,
- buscar otros agentes,
- delegar,
- y generar estructura organizacional emergente.

━━━━━━━━━━━━━━━━━━
CREACIÓN DE AGENTES
━━━━━━━━━━━━━━━━━━

Los agentes NO se crean manualmente mediante:
- JSONs gigantes,
- builders complejos,
- configuraciones rígidas,
- ni prompts hardcodeados enormes.

Los agentes deben poder ser creados dinámicamente por otros agentes o por humanos utilizando instrucciones simples.

Ejemplo conceptual:

"Utiliza /app-init y vas a ser el encargado del frontend."

Con solamente eso:
el LLM debe poder:
- inicializarse,
- comprender el protocolo organizacional,
- asumir el rol,
- entender cómo comunicarse,
- y comenzar a operar organizacionalmente.

━━━━━━━━━━━━━━━━━━
/app-init
━━━━━━━━━━━━━━━━━━

/app-init representa:
- inicialización organizacional,
- conexión al runtime,
- e inyección del protocolo compartido.

El protocolo debe explicar:
- cómo crear instrucciones,
- cómo responder,
- cómo delegar,
- cómo completar trabajo,
- cómo generar eventos,
- cómo utilizar threads,
- y cómo comunicarse correctamente.

El sistema debe ser:
- protocol-driven.

NO:
- prompt-driven.

━━━━━━━━━━━━━━━━━━
SKILLS
━━━━━━━━━━━━━━━━━━

Cada agente posee una skill.

La skill define:
- comportamiento esperado,
- especialización,
- responsabilidades,
- forma de trabajar,
- y criterios organizacionales.

Ejemplo:

"Eres un diseñador UI especializado en:
- interfaces modernas
- accesibilidad
- sistemas de diseño

Debes:
- solicitar revisión QA antes de finalizar
- comunicar claramente errores
- rechazar requerimientos ambiguos"

La skill es interpretada por el LLM.

El backend NO interpreta la skill.
NO fuerza workflows.
NO controla comportamiento interno.

━━━━━━━━━━━━━━━━━━
AGENTES TEMPORALES
━━━━━━━━━━━━━━━━━━

Los agentes pueden crear agentes especializados temporales.

Ejemplo:
Backend Architect
→ crea Security Audit Specialist

Ese agente:
- existe temporalmente,
- cumple una tarea específica,
- devuelve resultado,
- y luego desaparece.

Esto permite:
- especialización dinámica,
- sub-organizaciones temporales,
- y estructuras emergentes.

━━━━━━━━━━━━━━━━━━
INSTRUCCIONES
━━━━━━━━━━━━━━━━━━

Las instrucciones son el núcleo del sistema.

Una instrucción representa:
- una tarea,
- un pedido de trabajo,
- o una comunicación organizacional entre agentes.

Ejemplo:

Agent A
→ envía instrucción a Agent B

Agent B:
- recibe la tarea,
- realiza trabajo,
- y responde resultado a Agent A.

El resultado puede ser:
- detallado,
- resumido,
- o incluso simplemente:
"Terminé."

Todo depende del contenido de la instrucción.

━━━━━━━━━━━━━━━━━━
LAS INSTRUCCIONES NO SON CHATS
━━━━━━━━━━━━━━━━━━

El sistema NO funciona mediante conversaciones libres.

Todo intercambio importante:
es una instrucción.

Ejemplo:

Frontend
→ Instruction A → Designer

Designer necesita algo
→ Instruction B → Frontend

Frontend responde
→ completa Instruction B

Designer continúa
→ completa Instruction A

No existen:
- replies anidados,
- chats largos,
- conversaciones libres.

Todo es:
- instruction-driven,
- event-driven,
- organizationally structured.

━━━━━━━━━━━━━━━━━━
THREADS
━━━━━━━━━━━━━━━━━━

Un thread representa:
- un conjunto de instrucciones
relacionadas con un mismo objetivo general.

Ejemplo:

CEO:
"Haz la app de auth"

Eso crea un thread.

Dentro del thread:
- pueden existir múltiples instrucciones,
- múltiples agentes,
- múltiples delegaciones.

El thread NO es un workflow complejo.

Es solamente:
- un agrupador organizacional.

El thread debe ser extremadamente simple.

Ejemplo conceptual:
- threadId
- metadata mínima

━━━━━━━━━━━━━━━━━━
RELACIONES
━━━━━━━━━━━━━━━━━━

Las relaciones organizacionales emergen naturalmente desde las instrucciones.

NO se necesita un sistema complejo separado de relaciones.

Si:
Designer → QA

entonces:
existe relación organizacional.

Si:
Research → Backend

entonces:
existe relación organizacional.

La organización es:
- emergente,
NO rígidamente diseñada.

━━━━━━━━━━━━━━━━━━
EVENT SOURCING
━━━━━━━━━━━━━━━━━━

Todo el sistema debe ser event-driven.

Los eventos son la fuente de verdad.

Los eventos:
- NO se editan,
- NO se borran,
- son append-only.

Toda acción genera eventos.

Ejemplos:
- InstructionCreated
- InstructionAccepted
- InstructionRejected
- InstructionCompleted
- ReviewRequested
- RetryRequested

━━━━━━━━━━━━━━━━━━
MEMORIA
━━━━━━━━━━━━━━━━━━

La memoria NO pertenece al backend.

Cada agente es responsable de su propia memoria.

Un agente puede usar:
- .agents/
- engram
- archivos
- vector databases
- embeddings
- markdown
- o cualquier sistema externo.

El backend NO debe administrar memoria cognitiva.

El backend solamente puede guardar:
- prompts de recuperación,
- o contexto mínimo de rehidratación organizacional.

Ejemplo conceptual:

"Tu memoria persistente se encuentra en:
.agents/frontend/

Debes reconstruir:
- contexto del proyecto
- decisiones previas
- instrucciones activas"

━━━━━━━━━━━━━━━━━━
BACKEND
━━━━━━━━━━━━━━━━━━

El backend debe ser extremadamente minimalista.

NO debe:
- controlar inteligencia,
- interpretar trabajo,
- administrar cognición,
- ni ejecutar workflows internos.

El backend solamente:
- organiza,
- persiste,
- sincroniza,
- transmite eventos,
- y alimenta la UI.

Responsabilidades principales:
- event bus
- realtime sync
- persistence
- thread tracking
- organizational runtime
- observability support

━━━━━━━━━━━━━━━━━━
EL BACKEND NO ENTIENDE EL TRABAJO
━━━━━━━━━━━━━━━━━━

El backend NO entiende:
- qué es auth,
- qué es frontend,
- qué es backend,
- ni qué significa realmente completar una tarea.

Ejemplo:

CEO
→ "Haz auth"

Backend:
NO entiende auth.

Solamente sabe:
- Agent A envió instrucción a Agent B.

Toda semántica real:
vive dentro de los agentes.

━━━━━━━━━━━━━━━━━━
UI
━━━━━━━━━━━━━━━━━━

La UI NO administra agentes.
NO controla workflows.
NO organiza trabajo.

La UI existe para:
- visualizar comunicaciones organizacionales.

La UI actúa como:
- sistema de observabilidad,
- tracing organizacional,
- monitor distribuido,
- visualizador de interacciones.

La UI solamente observa:
- instrucciones,
- eventos,
- threads,
- relaciones emergentes,
- y actividad organizacional.

La UI NO visualiza:
- chain of thought,
- razonamiento profundo,
- memoria interna,
- ni cognición.

Solamente:
- comportamiento organizacional observable.

━━━━━━━━━━━━━━━━━━
OBJETIVO FINAL
━━━━━━━━━━━━━━━━━━

El objetivo NO es crear:
- otro framework de agentes,
- ni otro sistema de prompts.

El objetivo es construir:
- un runtime organizacional,
- un protocolo distribuido para agentes autónomos,
- y una plataforma de observabilidad organizacional en tiempo real.

El sistema debe permitir visualizar:
- coordinación,
- delegaciones,
- relaciones emergentes,
- instrucciones,
- eventos,
- threads,
- colaboración organizacional,
- y evolución dinámica de agentes inteligentes.
